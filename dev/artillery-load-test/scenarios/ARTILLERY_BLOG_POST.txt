서버를 개발하고 배포하기 전, 가장 두려운 순간은 언제일까? 바로 **"과연 이 서버가 실제 트래픽을 견딜 수 있을까?"**라는 의문이 들 때다.

단위 테스트(Unit Test)나 통합 테스트(Integration Test)는 로직의 정합성을 검증해주지만, **서버의 체력(Performance)**을 검증해주지는 않는다. 사용자가 10명일 때는 잘 돌아가던 코드가, 1,000명이 동시에 접속하면 DB 커넥션 풀이 말라버리거나 타임아웃이 발생하며 뻗어버릴 수 있기 때문이다.

오늘은 **Artillery**라는 강력하고 트렌디한 도구를 사용해서, 내 서버의 한계를 명확히 측정하고 최적화하는 과정을 아주 상세하게 공유하려고 한다.

---

## 1. 수많은 툴 중에 왜 Artillery인가?

부하 테스트 툴이라고 하면 보통 Java 기반의 `JMeter`를 많이 떠올린다. 하지만 JMeter는 GUI가 복잡하고 무겁다. 최근에는 Go 언어로 작성된 `k6`도 인기지만, JavaScript/TypeScript 생태계에 익숙한 Node.js 개발자에게는 **Artillery**가 압도적으로 편리하다.

*   **Node.js 기반**: `npm`으로 설치하고 실행한다.
*   **YAML 설정**: 복잡한 코딩 없이 YAML 파일 하나로 시나리오를 정의한다.
*   **Scenario-based**: 단순히 url을 때리는 게 아니라, "로그인 -> 검색 -> 상세 조회" 같은 사용자 흐름을 테스트한다.

## 2. 설치 및 기본 설정

설치는 아주 간단하다. 전역으로 설치해도 되고, 프로젝트의 `devDependencies`로 관리해도 된다.

```bash
npm install -g artillery
# 또는
npm install --save-dev artillery
```

## 3. Artillery 스크립트 한 줄 한 줄 뜯어보기 (.yml)

Artillery의 핵심은 YAML 파일이다. 가장 기본이 되는 피드 조회 시나리오(`feed.yml`)를 예시로, 각 설정이 무엇을 의미하는지 상세하게 설명한다.

```yaml
# mixed-scenario.yml

config:
  # 1. 타겟 서버 설정
  target: "https://api.my-service.com"
  
  # 2. 부하 단계(Phases) 설정
  phases:
    # 1단계: Warm-up (준비 운동)
    - duration: 60
      arrivalRate: 1
      rampTo: 1
      name: "Warm up"
    
    # 2단계: Peak Load (최대 부하)
    # 실제 서버가 견딜 수 있는지 확인할 목표치다.
    # 여기서는 초당 2명의 유저 진입(약 100~150명 동접 예상)을 설정했다.
    - duration: 100
      arrivalRate: 2
      rampTo: 2
      name: "Peak load"

  # 3. 플러그인 설정
  plugins:
    expect: {} # 응답 코드 검증을 위한 플러그인

scenarios:
  # 4. 가상 사용자 시나리오 정의
  - name: "User Journey - Active User"
    weight: 30 # 전체 유저 중 30% 확률로 실행
    flow:
      # (1) 리스트 조회
      - get:
          url: "/recipes/feed"
          expect:
            - statusCode: 200
      
      # (2) Think Time (중요!)
      # 사용자는 기계가 아니다. 3초간 고민한다.
      - think: 3

      # (3) 상세 조회 및 좋아요
      - get:
          url: "/recipes/123"
      - post:
           url: "/recipes/123/like"
```

### 핵심 포인트
*   `arrivalRate`: **초당 진입하는 사용자 수**다. 만약 평균 2분(120초) 머무르는 시나리오에서 `arrivalRate: 2`라면, 동시 접속자는 **240명**(2 * 120)에 육박하게 된다. 이 개념을 잘 잡아야 한다.
*   `think`: 실제 사용자와 유사한 패턴을 만들기 위해 필수적인 대기 시간이다.

---

## 4. 나의 시나리오 구성 전략

나는 단일 API 테스트에 그치지 않고, 실제 서비스 환경과 유사한 **Mixed Scenario(복합 시나리오)**를 구성했다.

*   `feed.yml`: 피드 조회 (Read Heavy)
*   `recipe-detail.yml`: 상세 조회 + 좋아요/댓글 (Read + Write)
*   `recipe-upload.yml`: 이미지 업로드 및 글쓰기 (Write Heavy)

이것들을 적절히 섞은 `mixed-scenario.yml`을 만들어, 조회하는 사람 50%, 좋아요 누르는 사람 30%, 검색하는 사람 20% 비율로 섞어서 테스트했다.

---

## 5. 트러블 슈팅: 실패에서 성공까지

### 1차 시도: 의욕이 앞선 설정 (실패) 🛑
처음에는 `arrivalRate: 10` (초당 10명 진입)으로 설정했다. "이 정도는 버티겠지?"라고 생각했다.
결과는 처참했다.

*   **Error**: `Timeout awaiting 'request' for 30000ms` (30초 타임아웃)
*   **Failures**: 700건 이상의 에러 발생
*   **Response Time**: 평균 **27초**

서버가 죽지는 않았지만, 요청이 큐에 쌓여 처리되지 못하는 **서비스 불능 상태**가 되었다. 리틀의 법칙으로 계산해보면 무려 동시 접속자 1,200명 수준의 부하였다. 단일 서버로는 무리였다.

### 2차 시도: 현실적인 목표 수정 (성공) ✅
목표를 수정했다. "동접 1,000명"이라는 막연한 목표 대신, **"동접 100명에서 안정적인가?"**를 먼저 검증하기로 했다.
`arrivalRate`를 1~2 수준으로 낮추고 다시 돌렸다.

```text
Summary report @ 23:41:25(+0900)
--------------------------------
http.codes.200: 1455
http.codes.201: 135
vusers.failed: 0 (실패 없음!)
http.response_time:
  median: 67.4 ms
  p95: 671.9 ms
  p99: 1085.9 ms
```

**결과 분석:**
*   **성공률 100%**: 단 한 건의 에러도 없이 모든 시나리오를 완주했다.
*   **쾌적한 속도**: 중앙값(Median)이 **67ms**로 매우 빠르다.
*   **안정성**: 하위 95%(p95)의 요청도 **0.67초** 안에 처리되었다.

이제 우리 서버는 **"동시 접속자 100~150명 수준에서는 0.1초 내외의 빠른 응답을 보장한다"**고 자신 있게 말할 수 있게 되었다.

---

## 6. 결론

부하 테스트를 통해 얻은 것은 단순히 "서버가 튼튼하다/아니다"가 아니었다.

1.  **숫자로 된 기준점**: 막연한 불안감이 "동접 150명까지 OK"라는 데이터로 바뀌었다.
2.  **한계 파악**: 무리하게 부하를 주면 어디서부터(DB인지, 앱인지) 병목이 오는지 경험했다.
3.  **자신감**: 이제 마케팅 팀이 "이벤트 해도 돼요?"라고 물으면 "네, 동접 100명까지는 끄떡없습니다!"라고 말할 수 있다.

개발자 여러분, 기능 구현이 끝났다고 끝난 게 아니다.
**사용자가 부수기 전에, Artillery로 먼저 부셔보자.** 그 과정에서 서버는 더 단단해진다.
